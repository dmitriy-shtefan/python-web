# Персональний проєкт

Протягом курсу кожен студент працює над власним персональним проєктом. 
Це не окрема другорядна активність, а головна практична лінія курсу: через цей проєкт студент 
поступово збирає всі ключові теми курсу в єдину робочу систему.

Підсумковий контроль проводиться у формі:

- захисту персонального проєкту;
- короткого теоретичного опитування за ключовими темами курсу.

Для допуску до підсумкового контролю мають бути виконані домашні та практичні завдання, 
а проєкт повинен мати всі обов'язкові проміжні результати.

## Для чого потрібен персональний проєкт

Персональний проєкт потрібен не для того, щоб зробити "великий стартап", а для того, щоб показати, що студент уміє:

- перетворити прикладну ідею на структуру сутностей і сценаріїв;
- спроєктувати Django-застосунок під конкретну предметну область;
- побудувати web-інтерфейс на основі routing, views і templates;
- організувати введення, перевірку та збереження даних;
- працювати з моделями, зв'язками, міграціями та Django admin;
- відкрити дані проєкту через базовий REST API;
- зібрати все це в цілісний, логічний, працездатний навчальний продукт.

## Яким має бути проєкт

Проєкт має бути навчальним, реалістичним за обсягом і таким, який можна довести до завершеного стану в межах курсу. 
Найкращий формат для цього курсу - невеликий, але завершений Django-проєкт з чіткою прикладною логікою.

Проєкт повинен мати:

- зрозумілу тему та предметну область;
- одну головну бізнес-ідею або практичну задачу;
- основну сутність і пов'язані з нею додаткові сутності;
- web-частину для роботи користувача;
- data layer на Django models;
- форми для введення і редагування даних;
- базовий API для частини даних проєкту.

## Що обов'язково має бути реалізовано

У персональному проєкті обов'язково мають бути такі складові:

- Сутності та зв'язки між ними.
- CRUD-операції: створення, читання, редагування, видалення.
- Django templates, базовий макет сторінок і навігація.
- Форми й валідація даних.
- Django models, міграції та робота з ORM.
- Django admin для керування даними.
- Авторизація: вхід, вихід, реєстрація або інший погоджений сценарій доступу користувача.
- Різні ролі користувача або різні типи дій для різних сценаріїв роботи.
- Базовий REST API для головної сутності або ключового набору даних.

## Мінімальний прийнятний масштаб

Щоб проєкт був достатнім для захисту, але не став надмірним, варто орієнтуватися на такий навчальний масштаб:

- 2-4 основні моделі;
- щонайменше 1 зв'язок між моделями;
- 1 ключова сутність, для якої реалізовано повний CRUD;
- список і детальна сторінка для головної сутності;
- сторінки створення і редагування даних через форми;
- авторизація та хоча б один сценарій, що залежить від користувача;
- 1-3 API-ендпоінти для демонстрації роботи з JSON.

Це не жорстка формальна норма, а орієнтир. Якщо проєкт простіший, він може не показати достатнього рівня володіння матеріалом. 
Якщо значно більший, є ризик, що студент не встигне якісно завершити його в межах курсу.

## Який формат теми підходить найкраще

Найкраще підходять теми, у яких є:

- об'єкти предметної області;
- користувацькі сценарії;
- дані, які потрібно створювати, змінювати, переглядати і видаляти;
- зрозуміла практична логіка;
- можливість показати зв'язки між сутностями.

Невдалими зазвичай є теми, які:

- занадто абстрактні і не мають нормальної моделі даних;
- зводяться лише до набору статичних сторінок;
- потребують надто складної зовнішньої інтеграції;
- за масштабом більше схожі на дипломний проєкт, а не на навчальний Django-MVP.

## Як проєкт розвивається протягом курсу

Персональний проєкт не робиться в останній тиждень. Він формується поетапно разом із проходженням модулів курсу.

### Етап 1. Вибір теми та предметної області

На початку курсу студент:

- обирає тему;
- формулює, яку практичну задачу розв'язує проєкт;
- визначає головну сутність;
- окреслює пов'язані сутності;
- продумує базові сценарії користувача.

Результат етапу: є короткий опис ідеї проєкту та чернетка предметної моделі.

### Етап 2. Старт Django-проєкту

Після знайомства з Django студент:

- створює проєкт і додаток;
- налаштовує базову структуру;
- підключає маршрути;
- запускає dev-сервер;
- створює перші view і сторінки.

Результат етапу: проєкт запускається, має базову структуру і перші робочі сторінки.

### Етап 3. Routing, views, templates

Під час модуля про routing і templates студент:

- створює базовий шаблон;
- налаштовує навігацію;
- реалізує список і детальну сторінку головної сутності;
- передає дані з view у template;
- підключає static files.

Результат етапу: проєкт уже має осмислений web-інтерфейс, а не лише технічну заготовку.

### Етап 4. Форми і сценарії введення даних

Під час модуля про форми студент:

- створює Django forms або ModelForms;
- реалізує введення даних;
- додає валідацію;
- реалізує сценарії create і update;
- коректно відображає помилки форми.

Результат етапу: у проєкті працює введення та редагування даних через web-інтерфейс.

### Етап 5. Models, ORM, міграції, admin

Під час модуля про models і ORM студент:

- формалізує моделі предметної області;
- створює та застосовує міграції;
- налаштовує зв'язки між сутностями;
- працює з ORM-запитами;
- реєструє моделі в Django admin.

Результат етапу: проєкт має цілісний data layer і придатний до нормальної роботи з даними.

### Етап 6. REST API

На завершальному технічному етапі студент:

- підключає Django REST Framework;
- створює serializer;
- налаштовує API-маршрути;
- реалізує базове API для головної сутності;
- перевіряє API через браузер, Postman або інший клієнт.

Результат етапу: проєкт показує не лише HTML-інтерфейс, а й базову API-складову.

### Етап 7. Інтеграція і підготовка до захисту

Перед захистом студент:

- перевіряє маршрутизацію, шаблони, форми, моделі та API;
- виправляє критичні помилки;
- готує короткий сценарій демонстрації;
- повторює ключові поняття курсу.

Результат етапу: проєкт зібраний у завершену навчальну систему, яку можна впевнено показати на захисті.

## Обов'язкові проміжні чекпоінти

Щоб проєкт не накопичувався на фінал, у ньому мають бути видимі проміжні результати.

### Чекпоінт 1. Після блоку Routing, views, templates

Має бути:

- запущений Django-проєкт;
- базова структура сторінок;
- навігація;
- список і детальна сторінка хоча б для однієї сутності;
- узгоджений зовнішній вигляд сторінок через шаблони.

### Чекпоінт 2. Після блоку Forms

Має бути:

- форма введення даних;
- форма редагування даних;
- валідація;
- коректна обробка помилок;
- робочий сценарій POST-запиту.

### Чекпоінт 3. Після блоку Models, ORM, admin

Має бути:

- завершена модельна основа проєкту;
- зв'язки між сутностями;
- міграції;
- наповнення і перевірка даних через admin;
- коректна робота CRUD-операцій з реальною базою даних.

### Фінальний технічний стан перед захистом

Має бути:

- web-частина проєкту;
- data layer;
- авторизація;
- базовий API;
- цілісна демонстрація основного сценарію роботи.

## Що саме перевіряється на захисті

На захисті оцінюється не "красивість" теми, а те, наскільки студент уміє застосувати матеріал курсу на практиці.

Перевіряється:

- чи проєкт запускається і працює;
- чи відповідає він заявленій темі;
- чи є в ньому моделі, зв'язки, форми, CRUD, шаблони й авторизація;
- чи логічно побудовані маршрути, views і шаблони;
- чи коректно організована робота з даними;
- чи може студент пояснити, як працює його код;
- чи може студент коротко показати API-частину;
- чи розуміє студент ключові терміни й механізми Django, які використав.

## Як варто будувати демонстрацію на захисті

Під час захисту студенту бажано показати проєкт у короткій, чіткій послідовності:

1. Назва проєкту і яку задачу він розв'язує.
2. Які сутності є в проєкті і як вони пов'язані.
3. Основний користувацький сценарій у web-інтерфейсі.
4. Створення або редагування даних через форму.
5. Дані в admin або коротке пояснення model layer.
6. Один або кілька API-ендпоінтів.
7. Відповіді на короткі теоретичні запитання.

## Рекомендації щодо вибору масштабу

Щоб проєкт був сильним, але керованим, варто:

- обирати тему, яку можна пояснити в 2-3 реченнях;
- не перевантажувати проєкт другорядними функціями;
- спочатку довести до ладу основний сценарій, а потім додавати покращення;
- не брати тему, де все тримається на складному фронтенді або сторонніх сервісах;
- думати в термінах навчального MVP, а не "ідеального продукту".

## Що вважається слабким проєктом

Слабким буде вважатися проєкт, у якому:

- немає чіткої предметної моделі;
- відсутні реальні зв'язки між сутностями;
- немає повного CRUD для головної сутності;
- форми формально присутні, але не працюють коректно;
- авторизація не реалізована або не інтегрована в логіку проєкту;
- API відсутній або зроблений лише номінально;
- студент не може пояснити архітектуру і власний код.

## Приклади вдалих тем

Нижче наведено приклади тем, які добре підходять для цього курсу:

- менеджер книжок або домашня бібліотека;
- каталог фільмів або серіалів з добірками;
- планувальник подій;
- менеджер завдань;
- система нотаток з категоріями;
- облік витрат і доходів;
- маленька система бронювання консультацій;
- каталог рецептів;
- трекер навчальних курсів;
- мінісистема обліку товарів і замовлень;
- блог або тематичний контент-портал;
- список вакансій або резюме-платформа у спрощеному вигляді.

## Приблизні приклади сутностей

Щоб швидше оцінити життєздатність теми, можна одразу думати такими зв'язками:

- бібліотека: книга, автор, жанр, добірка;
- планувальник: подія, категорія, місце, учасник;
- менеджер завдань: завдання, статус, проєкт, коментар;
- рецепти: рецепт, категорія, інгредієнт, позначка;
- витрати: транзакція, категорія, рахунок, користувач;
- вакансії: вакансія, компанія, відгук, навичка.

## Підсумок

Персональний проєкт у межах цього курсу - це завершений навчальний Django-проєкт, у якому студент демонструє вміння працювати з routing, templates, forms, models, ORM, admin, авторизацією та базовим REST API. Найкраща стратегія - обрати просту, але змістовну тему, послідовно виконувати чекпоінти і не відкладати основну роботу на фінальний тиждень.
